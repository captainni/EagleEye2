# 竞品分析详细参考标准

本文档提供竞品分析技能的详细分析标准、判定规则和注意事项。

## 为什么输出格式如此重要？

⚠️ **关键原因**：
1. **系统自动解析** - 输出将被后端自动解析为 JSON 对象并存入数据库
2. **格式错误会导致失败** - 任何额外的文字、markdown 标记都会导致解析失败
3. **中文引号问题** - 中文引号 `"` 和 `'` 不是有效的 JSON 字符，必须转义
4. **高亮匹配依赖** - keyPoints 字段内容必须与原文精确匹配才能实现高亮显示

---

## 一、字段详细说明

### 1. company（竞品公司）
**识别方法**：
- 从文章标题、导语或正文中识别
- 常见金融机构：平安银行、招商银行、工商银行、建设银行、农业银行、中国银行、交通银行、邮储银行、浦发银行、民生银行、中信银行、光大银行、华夏银行、兴业银行、广发银行等
- 如文章未明确提及，使用"未知竞品"

### 2. type（动态类型）

| 类型 | 说明 | 判断依据 |
|------|------|----------|
| **产品更新** | 新产品发布、功能升级、产品迭代 | "推出...产品"、"发布...功能"、"上线..." |
| **营销活动** | 促销活动、市场推广、客户活动 | "开展...活动"、"推出...优惠"、"举办..." |
| **财报数据** | 业绩发布、数据披露、季报年报 | "发布...财报"、"业绩显示..."、"净利润..." |
| **APP更新** | 移动端功能优化、APP版本升级 | "APP升级"、"上线新功能"、"优化..." |
| **利率调整** | 存贷款利率变化、费率调整 | "利率下调"、"优惠利率"、"年化..." |
| **合作动态** | 战略伙伴、投资并购、业务合作 | "合作"、"战略合作"、"投资..." |
| **政策响应** | 对监管政策的响应措施、合规调整 | "响应政策"、"落实通知"、"合规..." |

### 3. importance（重要程度）

| 级别 | 说明 | 判断标准 |
|------|------|----------|
| **高** | 涉及核心产品/重大战略、直接影响市场竞争格局、可能改变行业态势 | 产品创新、重大合作、政策响应、财报数据 |
| **中** | 一般业务动态、有一定影响但非核心 | 营销活动、功能优化、常规合作 |
| **低** | 常规活动、影响有限 | 小型活动、常规更新、内部调整 |

### 4. relevance（与我方产品的相关度）

**有产品信息时**：
- **高**：竞品动态直接涉及我方核心业务或主要产品（如都是消费贷产品）
- **中**：竞品动态与我方有一定关联，但不是直接影响（如同属零售金融但产品不同）
- **低**：竞品动态与我方关联度较低（如对公业务 vs 零售业务）

**无产品信息时**：
- 默认为"中"

### 5. tags（标签）
- 提取 2-5 个相关领域关键词
- 常见标签：信用修复、政策响应、零售信贷、财富管理、数字化转型、普惠金融、风险管理、客户运营、消费贷、信用卡、理财、按揭等
- 标签应简洁、准确、便于分类

### 6. summary（摘要）
- 长度：200-300 字
- 内容：概括动态背景、主要内容、举措、潜在影响
- 风格：简洁明了，避免复制粘贴大段原文

### 7. keyPoints（关键要点 - ⚠️ 重要）
- 提取 3-5 条需要特别注意的关键信息
- **必须是原文语句，不得臆造或改写**
- 这些要点将用于在原文中**高亮显示**
- 每条要点应该是完整的句子或短语，便于精确匹配
- 选择对业务有价值、值得关注的内容

### 8. marketImpact（市场影响分析）
- 分析该动态对行业/市场的整体影响
- 评估是否会引起其他竞品跟进
- 识别潜在的市场机会或威胁
- 视角：行业整体、市场趋势

### 9. competitiveAnalysis（竞争态势分析）
- 分析此举对竞争格局的影响
- 评估对我方的直接威胁或机会
- 识别竞品优势和我们的差距
- **视角：站在我方立场分析**

### 10. ourSuggestions（可执行建议）
- 提出 2-3 条可执行、可落地的建议
- 每条建议需说明理由和依据
- 建议应具体明确，避免泛泛而谈
- **聚焦于"我方应该如何应对"**

---

## 二、注意事项

### 关键规则
1. ✅ **keyPoints 必须是原文语句** - 用于精确匹配和高亮，这是核心功能
2. ✅ **所有分析必须基于文章内容** - 不得臆造信息
3. ✅ **建议要具有可操作性** - 避免空泛的建议
4. ✅ **competitiveAnalysis 要站在我方视角** - 分析对我方的影响
5. ✅ **ourSuggestions 要聚焦行动** - 回答"我们应该如何应对"

### 常见错误
- ❌ keyPoints 使用概括性文字而非原文语句
- ❌ 添加文章中没有的信息
- ❌ 建议过于宽泛（如"加强关注"）
- ❌ 忽视我方立场，泛泛而谈

---

## 三、示例对比

### 优秀的 keyPoints（原文语句）
```
"成立由零售信贷风险管理部牵头的'一次性信用修复政策工作小组'"
"口袋银行APP上自主查询本机构逾期信息功能也已投产验证"
"坚决抵制'征信洗白'等非法中介骗局"
```

### 不好的 keyPoints（概括性文字）
```
"成立专项工作小组"  // 太宽泛，无法精确匹配
"APP功能升级"  // 不够具体
"加强风险管理"  // 原文中没有
```

## 四、快速参考

### 动态类型快速判断
- 看到"推出/发布/上线" → 产品更新
- 看到"活动/优惠/促销" → 营销活动
- 看到"财报/业绩/季报" → 财报数据
- 看到"APP升级/版本更新" → APP更新
- 看到"利率/年化/优惠" → 利率调整
- 看到"合作/战略/投资" → 合作动态
- 看到"响应政策/落实通知" → 政策响应

### 重要性快速判断
- 有"首个/首次/率先/创新" → 高
- 有"合作/战略/投资" → 高
- 常规营销/功能优化 → 中
- 内部调整/小型活动 → 低

### 相关度快速判断
- 相同产品线 → 高
- 相关业务线 → 中
- 不同业务线 → 低
